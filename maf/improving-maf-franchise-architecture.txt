2) OpenProse = “Declarative orchestration of long-running sessions”
OpenProse frames itself as a programming language for long-running AI sessions, with explicit control flow (parallelism, loops, try/catch, etc.) and a Claude Code plugin install path. OpenProse+1
But it also warns: Telemetry on by default and “Not for production”. OpenProse
In MAF terms: OpenProse is useful as a workflow DSL (a readable “playbook format”), not as the thing that decides what is safe.

The “most autonomous + most reliable” way to use them together
Principle: Control plane is strict; workflow plane can be expressive


BAML governs “what actions are allowed + what must be true to proceed.”


Prose describes “how to run a multi-step flow once actions are allowed.”


So you get the best of both:


deterministic contracts + gates (BAML)


readable workflows (Prose)


Critical rule for reliability:
Avoid OpenProse’s semantic completion checks for critical stuff (e.g. loop until **production ready**) because that’s judgment-based by design. GitHub+1

subtree health check passes


lint passes


PR created


required evidence exists


And if you still want “AI judgment,” route it through a BAML “Judge” function that outputs a structured verdict + rationale + required fixes.
