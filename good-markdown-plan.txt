What makes for a good markdown plan document for a software development project? What's the difference between a good plan and a great one? I'm always going on and on about how I spend 85%+ of my time and energy on the planning phases. Well what does that entail exactly? It's hard to explain in the abstract; you need a tangible example to really illustrate the nuances. So I thought I would share a good example from today. This also addresses a recent question I've been getting about my approach. People seem to be under the impression that you have to do everything in your project in one shot. And in my approach, that is true, but only for version 1! If you decide you want to add new features or change how things work, you can obviously do that once you have a functioning v1. And the way you do that is the same way you create v1, by creating a super detailed markdown plan first and then turning it into beads. It offers sub 50ms instant "search as you type" across all those logs and has many other nice features. I decided that I'd like to add a feature to Cass that is similar to a feature I already have in MCP Agent Mail and in beads_viewer (bv): the ability to export your setup as a static website that can be served using GitHub Pages. You can see an example for bv for this very project, which is the final result of the planning process I will describe in this post: [https://dicklesworthstone.github.io/beads_for_cass/](https://t.co/iw50xoFBgd) This functionality makes it very quick and easy to generate and deploy the exported site using the gh utility. But GitHub Pages, as nice as it is, only works for public repos (btw, my tools also support Cloudflare pages, but GH Pages is better and easier for this use case). So how to handle these problems? The answer is encryption: the user first selects which coding agents to include, which project folders, the time period, etc. and a bundle is generated (note that this bundle is in the canonical format that Cass internally converts all coding agent messages into from their original native formats) and then the user provides a password to use for the encryption of that bundle. Once the password is entered, it would unlock a beautiful, responsive UI that lets you easily search across the messages nearly as quickly and efficiently as Cass does. And if you really have nothing to hide, you can leave the password off and make it all actually public. Anyway, this is a super complex new feature to add for many reasons. A good markdown plan doc is not “a long brainstorm.” It’s an executable spec for V1: a document that lets a (possibly amnesiac) coding agent produce correct code with minimal back-and-forth, and lets you review correctness before you pay the implementation cost.
What a good markdown plan document does
A good plan removes ambiguity on the things that cause rework:


Pins down V1


Goals (what success looks like)


Non-goals (what you’re explicitly not doing yet)


Constraints (security, offline-only, portability, perf budgets, etc.)




Describes user workflows as primary truth


Step-by-step “what the user does” flows


Inputs/outputs, error states, empty states


UX expectations (what’s fast, what’s safe, what’s reversible)




Defines the “system contract”


Data formats, schemas, file layouts, versioning


Module boundaries and responsibilities


CLI/API surface + flags + semantics


Backwards compatibility / migration promises




De-risks the hard parts on paper


Threat model + privacy model (esp. if encryption/sharing)


Failure modes and recovery strategies


Performance targets + measurement approach


Platform gotchas (browser limits, wasm, GH Pages quirks, etc.)




Ends with an implementation path


A task graph that naturally becomes beads (epics → tasks → subtasks)


Acceptance criteria per task (“done means X is verifiably true”)


Test plan baked in (unit + integration + e2e), not “later”




That’s the core of “spending 85% on planning”: you’re moving unknowns and complexity left—into a space where edits are cheap.

Good vs Great: the real difference
Good plan = “I can implement V1 without guessing the main architecture.”
Great plan = “I can implement V1 without guessing edge cases, invariants, security posture, performance posture, and rollback paths.”
Here are the “greatness multipliers”:
1) It’s falsifiable
A great plan is full of statements that can be proven true/false:


“Exported site unlocks only with correct password”


“Search results update under 50ms for N records on typical laptop”


“No decrypted plaintext is written to disk or localStorage”


“Bundle format is forward-compatible via schema versioning”


2) It names invariants (and guards them)
Great plans explicitly say:


What must never happen (e.g., plaintext leak, key persistence)


What must always happen (e.g., authenticated encryption, integrity checks)


What’s acceptable degradation (e.g., slower first-load OK, typing must stay fast)


3) It’s risk-ordered
Good plans list tasks in a reasonable order.
Great plans put “unknowns” first (crypto choices, wasm/sqlite feasibility, GH Pages packaging constraints), so you fail fast in planning/prototyping rather than late in integration.
4) It’s “beads-native”
A great plan reads like it wants to become beads:


Each section maps cleanly to a bead cluster


Dependencies are explicit


Acceptance criteria and tests are attached to the bead that creates the behavior

markdownCopy code# PLAN: GH Pages Web Export (Encrypted) for cass

## Goal
Add a new export mode that generates a static website view of selected cass sessions, deployable to GitHub Pages. The export can be public or password-protected via encryption.

## Non-goals (V1)
- Multi-user accounts / auth server
- Sharing individual messages via deep links across bundles
- Collaborative annotations

## User Workflow
1) User runs `cass export web` and selects:
   - agent(s), project folder(s), time range
   - optional password (empty = no encryption)
2) cass generates an export folder:
   - static web UI assets
   - data bundle (encrypted if password provided)
3) User can run `cass export web --deploy gh-pages` to publish via `gh`.

## UX Requirements
- If encrypted: page shows password prompt, no data visible until unlocked.
- After unlock: responsive UI, fast search, filters by agent/project/time.
- If unencrypted: site loads immediately.

## Data Bundle
- cass exports in canonical cass schema.
- Bundle includes a manifest with metadata and schema version.
- Encrypted bundle uses authenticated encryption.

## Implementation Outline
